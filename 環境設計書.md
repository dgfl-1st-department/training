# 環境設計書

## 1. 概要

本設計書は、アプリケーションの開発および実行環境に関するシステム構成、ミドルウェア、ネットワーク設定、および各コンポーネントの役割について定義する。
本システムを構築するにあたり、「基本設計書」および「要件定義書」の定義内容を満たす具体的な環境構成を示すことを目的とする。

## 2. システム構成概要

本システムは Docker および Docker Compose を用いてコンテナ化されており、以下の4つの主要サービスで構成される。

- **Frontend (フロントエンド)**: React + Vite による SPA アプリケーション
- **Backend (バックエンド)**: FastAPI + Python (uv 依存管理) による REST API サーバー
- **DB (データベース)**: MySQL 8.0 によるデータストア
- **Nginx (Webサーバー/リバースプロキシ)**: クライアントからのリクエストをフロントエンドとバックエンドへ振り分ける

## 3. ディレクトリ構成

```text
.

│   ├── public/           # 静的ファイル群

│   ├── vite.config.ts    # Viteのビルド設定

│   ├── main.py           # FastAPIのアプリケーションエントリポイント
│   ├── pyproject.toml    # Pythonパッケージ依存関係の定義 (uv用)
│   ├── uv.lock           # 依存パッケージのロックファイル
│   └── Dockerfile        # バックエンドコンテナのビルド定義
├── db/                   # データベース関連
│   └── init/             # DB初期化スクリプト (起動時に自動実行されるSQL等)
├── nginx/                # リバースプロキシ関連
```

## 4. コンポーネント別環境詳細

### 4.1 Frontend (フロントエンド)

- **言語**: TypeScript
- **フレームワーク・ライブラリ等**: React 18.x, Vite, Node.js
- **ビルド・実行環境**: `./frontend/Dockerfile` によるコンテナビルド
- **公開ポート**: `5173` (コンテナ内 `5173` とマッピング)
- **ボリュームマウント**:
  - `./frontend:/app` (ソース変更のホットリロード時同期用)
  - `/app/node_modules` (コンテナ内の Node モジュール保持用)
- **依存関係**: バックエンドサービス (`backend`) の起動に依存

### 4.2 Backend (バックエンド)

- **言語**: Python
- **フレームワーク・ライブラリ等**: FastAPI 0.100+, パッケージマネージャ `uv`
- **ビルド・実行環境**: `./backend/Dockerfile` によるコンテナビルド
- **公開ポート**: `8000` (コンテナ内 `8000` とマッピング)
- **ボリュームマウント**: `./backend:/app` (ソース変更のホットリロード時同期用)
- **環境変数**: ルートの `.env` ファイルを用いて DB 接続情報等を読み込み
- **依存関係**: DBサービス (`db`) のヘルスチェック成功 (`service_healthy`) に依存

### 4.3 DB (データベース)

- **ベースイメージ**: `mysql:8.0`  (コンテナ名: `mysql_db`)
- **公開ポート**: `3306` (コンテナ内 `3306` とマッピング)
- **環境変数設定**:
  - データベース名: `webapp_db` (`MYSQL_DATABASE`)
  - DB管理者(root)パスワード: 環境変数 `$DB_PASSWORD` によって設定
- **ボリュームマウント**:
  - `mysql_data:/var/lib/mysql` (データ永続化用の名前付きボリューム)
  - `./db/init:/docker-entrypoint-initdb.d` (起動時の初期化スクリプト自動実行用)
- **ヘルスチェック設定**: `mysqladmin ping` コマンドにより起動状態を5秒間隔で監視

### 4.4 Nginx (Webサーバー/リバースプロキシ)

- **ベースイメージ**: `nginx:latest`
- **公開ポート**: `80` (コンテナ内 `80` とマッピングされており、エンドユーザーの入口となる)
- **ボリュームマウント**: `./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro` (読取専用で設定をマウント)
- **依存関係**: Frontend (`frontend`) および Backend (`backend`) アプリケーションの起動に依存してルーティングを行う

## 5. ネットワークとデータ永続化

### 5.1 ネットワーク通信

Docker Compose のデフォルトネットワーク機能により、各コンテナ間はコンテナサービス名 (`backend`, `frontend`, `db`, `nginx`) による相互名前解決および通信アクセスが行われる。
具体的な内部トラフィックのルーティングや通信の流れは以下の通りである。

- **クライアント → Nginx (リバースプロキシ)**
  クライアント（ブラウザ）からの HTTP リクエストは、Nginx コンテナ（公開ポート `80`）が代表してすべて受け付ける。
- **Nginx → フロントエンド (SPA)**
  `/` に対するリクエスト等は `frontend` コンテナ（内部ポート `5173`）へ転送され、React (Vite) アプリケーションが配信される。
- **Nginx → バックエンド (API)**
  `/api/`, `/docs`, `/openapi.json` へのリクエストは `backend` コンテナ（内部ポート `8000`）へ転送され、FastAPI が処理を行う。
- **バックエンド → データベース**
  APIの動作に必要なデータ操作は、`backend` コンテナから `db` コンテナ（内部ポート `3306`）へ直接接続して行われる。

### 5.2 データ永続化 (Volumes)

システムデータの永続化手段として、名前付きボリューム (`mysql_data`) を定義している。
基本設計で定義されているユーザー情報、回答データ、セッション、監査ログなどは MySQL に保存され、コンテナの再起動・再作成後も本ボリュームを介して保持される。

## 6. 起動および動作前提

1. **動作環境**: Docker Desktop 等、Docker および Docker Compose V2 対応環境が整備されていること。
2. **環境変数の準備**: 起動前に、リポジトリルートに `.env` ファイルを作成し、`.env_example` を参考に必要な環境変数情報 (`DB_PASSWORD` や `DATABASE_URL` など) を設定すること。
3. **起動手順**: ターミナルから `docker compose up -d --build` コマンドを実行し、バックグラウンドですべてのコンテナが Health となることを確認してサービス利用を開始する。
（※初回起動時や、変更があった場合は `--build` を指定し、明示的なビルドが不要な場合は外して起動する）
4. **動作確認**:
   - **フロントエンド (SPA)**: ブラウザで `http://localhost` (もしくは直接 `http://localhost:5173`) にアクセスし、ログイン画面等が表示されることを確認する。
   - **バックエンド (API)**: ブラウザで `http://localhost/docs` (もしくは直接 `http://localhost:8000/docs`) にアクセスし、APIドキュメントが閲覧できることを確認する。
5. **停止手順**: ターミナルから `docker compose down` コマンドを実行し、すべてのコンテナおよびネットワークを停止・削除する。
   - ※ データベースのデータ（ボリューム）も合わせて初期化（完全に削除）したい場合は、`docker compose down -v` を実行する。（通常時はデータ保持のため `-v` は付与しない）

## Issue作成ルール

本プロジェクトは[Issue駆動開発](https://qiita.com/NaaaRiii/items/2e302283f96d2b0ee501)を採用する。

### テンプレート

```text
## 概要
（何を実装・対応するか、1〜2文で要約）

## 要件
- （実装すべき内容を箇条書き）
- （設計書や仕様に基づいた具体的な条件）

## 完了条件
- [ ] （完了とみなすためのチェック項目1）
- [ ] （完了とみなすためのチェック項目2）
```

### 各項目の説明

| 項目 | 説明 |
|------|------|
| **概要** | 課題の目的を簡潔に記載。何を誰向けに実装するかを明確にする |
| **要件** | 実装すべき機能・仕様を箇条書きで列挙。要件定義書・基本設計書を参照する |
| **完了条件** | 完了とみなすための検証可能なチェック項目。PR マージ前にすべて満たすこと |

### Issue のルール

- **1 Issue 1 機能**: 1つの Issue で1つの機能・修正に絞る
- **完了条件は検証可能に**: 曖昧な表現を避け、動作確認で判定できる内容にする
- **関連 Issue**: 依存関係がある場合は Issue 番号でリンクする
- **例外**: ドキュメントのみの軽微な更新（誤字修正、ルール追記、リンク修正など）は Issue 不要とする

### 作成例

```text
## 概要
未認証ユーザー向けのログイン画面UIを実装する。

## 要件
- アプリ名（またはロゴ）と「Googleでログイン」ボタンを配置
- 未認証時のアクセス先はすべてこの画面にリダイレクトさせる
- ログイン成功後、ユーザーのロール（employee / admin）に応じて初期画面へ遷移させる

## 完了条件
- [ ] 画面レイアウトが実装されていること
- [ ] ボタンクリックでGoogle OAuth画面へ正しく遷移すること
- [ ] ログイン完了後、権限に応じた正しい画面へ遷移すること
```

## ブランチ戦略

本プロジェクトでは[GitHub Flow](https://docs.github.com/ja/get-started/using-github/github-flow)を採用する。

### ブランチ構成

| ブランチ | 説明 |
|---------|------|
| `main` | 本番環境にデプロイ可能な安定版。常にデプロイ可能な状態を保つ |
| `feature/*` | 機能開発用。`feature/#<Issue番号>-実装内容` の形式 |

### 運用ルール

1. **main からの分岐**: 新規開発は必ず `main` から `feature/*` ブランチを作成する
2. **PR 経由のみ**: `main` への変更は Pull Request を経由する（詳細は[ブランチ保護](#ブランチ保護main-ブランチ)を参照）

### ブランチ保護（main ブランチ）

`main` ブランチには以下の保護ルールを適用します（GitHub の Branch protection rules で設定）。

| 設定項目 | 内容 |
|---------|------|
| 強制 push 禁止 | `git push --force` による上書きを禁止。履歴の改変を防ぐ |
| 直接 push 禁止 | `main` への直接 push を禁止し、PR 経由のみマージを許可 |
| 削除保護 | `main` ブランチの削除を禁止 |

### マージ条件

PR を `main` にマージするには、以下の条件をすべて満たすこと。

| 条件 | 内容 |
|------|------|
| Approve | 最低1名以上の Approve を取得 |
| レビュー対応 | 指摘事項にすべて対応（または合意を得ていること） |
| コンフリクト解消 | `main` とのコンフリクトを解消 |

### ブランチ作成手順

```bash
git checkout main
git pull origin main
git checkout -b "feature/#1-add-login-screen"
```

> **補足**: ブランチ名に `#` を含める場合は、シェルでコメント扱いされないよう引用符で囲む。

### ブランチ名作成のTips

| ポイント | 説明 |
|---------|------|
| **Issue 番号を必ず含める** | `#1` を含めると PR と Issue が紐づく。`#` を使う場合は引用符で囲む |
| **ケバブケース** | 単語はハイフンで区切る（例: `login-screen`、`add-oauth`） |
| **小文字で統一** | 大文字は使わず、すべて小文字にする |
| **短く具体的に** | 30文字以内を目安に、何の機能か分かる名前にする |
| **動詞で始める** | 機能名は動詞で始めると意図が明確になる。例: `add-login-screen`、`fix-date-validation` |
| **スラッシュ・スペース禁止** | ブランチ名に `/` やスペースは使わない (feature/の区切りは除く) |

#### よく使われる動詞の例

| 動詞 | 意味 |
|------|------|
| `add` | 追加 |
| `implement` | 実装 |
| `fix` | 修正 |
| `create` | 作成 |
| `update` | 更新 |
| `refactor` | リファクタ |
| `remove` | 削除 |
| `improve` | 改善 |

#### 良い例

```text
feature/#1-add-login-screen
feature/#2-implement-oauth
feature/#15-fix-date-validation
```

#### 避けるべき例

```text
feature/login           # Issue 番号がない
feature/#1-LoginScreen  # キャメルケース（ケバブケース推奨）
feature/#1-add-function # 曖昧で内容が分からない
feature/#1-my-branch    # 個人名・汎用的な名前
```

## コミットルール

### コミットメッセージ形式

```text
<Type>: <Title>

- [Details]
```

### Type（必須）

| Type | 説明 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントの変更（README、設計書など） |
| `style` | コードの意味に影響しない変更（フォーマット、空白など） |
| `refactor` | リファクタリング |
| `test` | テストの追加・修正 |
| `chore` | ビルド、設定ファイル、依存関係の変更 |

### コミットのルール

- **Title**: #<Issue番号> 変更内容
- **Details**: 一行空けて箇条書きする(必須ではなく必要に応じて記載)
- **日本語で記載**: Title、Detailsともに日本語で記載する
- **1コミット1変更**: 1コミットで1つの変更に留める
  - 原則として Issue に対応しないコミットを作らない
  - 例外: ドキュメントのみの軽微な更新（誤字修正、ルール追記、リンク修正など）は Issue 不要

### コミット例

```text
feat: #1 ログイン画面のUIを追加

- ログイン画面に○○を追加
```

## PR作成ルール

### 作成前の確認

- [ ] 対応 Issue の[完了条件](#各項目の説明)をすべて満たしている（[ドキュメントのみの更新は例外](#issue-のルール)）
- [ ] `main` を最新化し、[マージ条件](#マージ条件)を満たせる状態にしている
- [ ] ローカルで動作確認・テストが通っている
- [ ] コミットメッセージがルールに従っている

### PR のタイトル

- **タイトル**: `#<Issue番号> 実装内容(Issue名でも良い)`

### PR の説明（テンプレート）

Issue 駆動開発に合わせ、対応 Issue とその完了条件を明示する。

```text
## Issue
closes #<Issue番号>

## 完了条件の確認
（対応 Issue の完了条件をコピーし、確認済みのものは [x] にする）
- [ ] （Issue の完了条件1）
- [ ] （Issue の完了条件2）

## 変更内容
- （箇条書きで変更点を列挙。Issue の要件との対応が分かるとよい）

## その他
- [ ] 動作確認済み
- [ ] ドキュメントの更新が必要な場合は対応済み
```

### PR 作成例（Issue #1 ログイン画面の実装）

```text
## Issue
closes #1

## 完了条件の確認
- [x] 画面レイアウトが実装されていること
- [x] ボタンクリックでGoogle OAuth画面へ正しく遷移すること
- [x] ログイン完了後、権限に応じた正しい画面へ遷移すること

## 変更内容
- ログイン画面コンポーネントを追加
- Google OAuth 認証フローを実装
- 未認証時のリダイレクト処理を追加

## その他
- [x] 動作確認済み
- [x] ドキュメントの更新が必要な場合は対応済み
```

### レビュー手順

GitHub の PR レビュー機能を用いて、以下の手順で行う。

#### PR 作成者

1. **レビュー依頼**: PR 画面の「Reviewers」からレビュアーを指定し、レビューを依頼する
2. **指摘への対応**: レビューコメントに対して返信し、修正した場合は「Resolve conversation」で解決済みにする
3. **再レビュー依頼**: 修正後、必要に応じて再度レビューを依頼する

#### レビュアー

1. **Files changed で確認**: PR の「Files changed」タブで差分を確認する
2. **コメントで指摘**: 指摘箇所にカーソルを合わせ「+」をクリックし、行コメントを追加する。指摘は「何が」「なぜ」「どう直すか」を具体的に記載する
3. **レビュー提出**: 確認後、「Review changes」から以下を選択して提出する
   - **Comment**: コメントのみ（マージ可否は示さない）
   - **Approve**: 承認。マージ可能と判断した場合
   - **Request changes**: 変更を求める。指摘がある場合
4. **再確認**: 修正後、再度「Files changed」で差分を確認し、指摘が解消されていれば Approve する

#### 確認観点

- PR のタイトル・説明がルールに従っているか
- 対応 Issue の完了条件がすべて満たされているか
- コードの品質（可読性、命名、責務の分離）が適切か
- 設計書・コーディング規約に沿っているか

### マージ

- [マージ条件](#マージ条件)を満たしたうえでマージする

### マージ後

- `feature/*` ブランチを削除する
